01 Meaningful Names

Names are the most important thing you write. A good name is a free comment — it explains itself without any extra words.

The Rule Use snake_case for variables and functions. Names should say what something is, not what type it is.
✗ Messy hard to understand at a glance 
def calc(a, b, c):
    return a + b * c   # what are a, b, c?

x = 10             # what is x?
myList1 = []      # type in the name → avoid
boolFlag = True   # redundant suffix
✓ Clean reads like plain English 
def calculate_total_price(price, quantity, tax_rate):
    return price + quantity * tax_rate

age = 10
students = []
is_logged_in = True
02 Small, Focused Functions

Each function should do one thing and do it well. If you can't describe it in a sentence without "and", split it up.

The Rule Aim for functions under ~15 lines. If a function is doing input, processing, AND output — break it into three functions.
✗ Messy one function doing everything 
def do_everything(name, score):
    # validate
    if score < 0 or score > 100:
        print("Invalid score")
        return
    # calculate grade
    if score >= 90: grade = "A"
    elif score >= 70: grade = "B"
    else: grade = "C"
    # print result
    print(f"{name} scored {score} — Grade: {grade}")
✓ Clean each function has one job 
def is_valid_score(score):
    return 0 <= score <= 100

def get_letter_grade(score):
    if score >= 90: return "A"
    if score >= 70: return "B"
    return "C"

def display_result(name, score):
    if not is_valid_score(score):
        print("Invalid score")
        return
    grade = get_letter_grade(score)
    print(f"{name} scored {score} — Grade: {grade}")
03 Comments That Add Value

Comments should explain why, not what. If your code is clear, it explains itself. Comments are for the parts that can't.

✗ Messy comments that repeat the code 
# add 1 to counter
counter += 1

# loop through users
for user in users:
    # print user name
    print(user.name)
✓ Clean comments that explain the "why" 
counter += 1  # track retries before timeout

# API returns users sorted by signup date (oldest first),
# but we display newest first — reverse before rendering
for user in reversed(users):
    print(user.name)
Docstrings for Functions Add a short docstring to any function that isn't immediately obvious. Use triple quotes.

def calculate_discount(price, percent):
    """Return discounted price. Percent should be 0–100."""
    return price * (1 - percent / 100)
04 Spacing & Formatting (PEP 8)

Python has an official style guide called PEP 8. You don't need to memorise it — just know the core rules.

✗ Messy inconsistent whitespace 
x=5+3
def greet( name,greeting="Hello" ):
    print(greeting+","+name)
greet("Alice","Hi")
✓ Clean PEP 8 spacing 
x = 5 + 3

def greet(name, greeting="Hello"):
    print(greeting + ", " + name)

greet("Alice", "Hi")
Quick Spacing Rules Spaces around operators: x = 5 not x=5. No spaces inside parentheses. Two blank lines between top-level functions. Use 4 spaces (not tabs) for indentation.
05 Cleaner Conditionals

Long chains of if/else are hard to read. Python has elegant shortcuts worth knowing early.

✗ Messy verbose comparisons 
if name == "Alice" or name == "Bob" or name == "Carol":
    print("Admin")

if is_active == True:
    send_email()

result = None
if score >= 50:
    result = "pass"
else:
    result = "fail"
✓ Clean pythonic conditionals 
admins = {"Alice", "Bob", "Carol"}
if name in admins:
    print("Admin")

if is_active:          # don't compare to True
    send_email()

result = "pass" if score >= 50 else "fail"
06 List Comprehensions

One of Python's best features. Turn multi-line loops into single readable expressions — but only when it stays clear.

✗ Verbose loop-based list building 
even_squares = []
for n in range(10):
    if n % 2 == 0:
        even_squares.append(n ** 2)
✓ Pythonic list comprehension 
even_squares = [n ** 2 for n in range(10) if n % 2 == 0]
When Not to Use Them If the comprehension spans more than one line or has nested logic, use a plain loop. Readability wins.
07 Handling Errors Properly

Catching errors is good. Catching all errors silently is dangerous — it hides bugs and makes debugging a nightmare.

✗ Messy silent blanket catch 
try:
    result = 10 / user_input
except:          # catches EVERYTHING, even typos
    pass         # error disappears silently
✓ Clean specific, informative error handling 
try:
    result = 10 / user_input
except ZeroDivisionError:
    print("Error: cannot divide by zero.")
    result = None
except TypeError as e:
    print(f"Unexpected input type: {e}")
    result = None
✓ Clean Code Checklist

Before you consider any script "done", run through this list.

  • ✓ Variable names describe what they hold
  • ✓ Functions do one thing each
  • ✓ No function longer than ~20 lines
  • ✓ Comments explain why, not what
  • ✓ 4-space indentation, consistent throughout
  • ✓ Spaces around all operators
  • ✓ No magic numbers — use named constants
  • ✓ Errors caught specifically, not silently
  • ✓ No copy-pasted code blocks (extract a function)
  • ✓ You can read it aloud and it makes sense
